**2. Documentation-sync bug — found, root-caused, and fixed in two escalating passes (PR #313, #314).**
`COWORK_HANDOFF.md`, `OPEN_ISSUES.md`, and `CLAUDE_CODE_STATUS.md` were discovered to be carrying stale "not yet delivered" content despite PR #312 being fully merged and deployed. First pass (PR #313) fixed what looked like a shallow staleness in one section. During that PR's merge, a real conflict surfaced showing the fix branch had been cut from a commit *before* PR #312 — proof the staleness was real, not just a local read artifact. Root cause, confirmed conclusively via `git reset --hard origin/main`: the actual git-committed content of these three files (not just a local read) was an intermediate, pre-delivery snapshot — a genuine gap between when file edits land on the OneDrive-synced disk and when a subsequent `git add` captures those bytes. Second pass (PR #314) corrected the deeper issue and recorded accurate backfill numbers.

Separately discovered and worked around: `git pull` was silently failing to actually update local `main` all session, because one unrelated pre-existing remote branch with an extremely long name causes `git fetch` to exit non-zero (`Filename too long` trying to create its lock file), which aborts `pull`'s merge step before `main` gets updated — even though `pull`'s output looks like it succeeded. Established workaround for future sessions: `git fetch origin main` + `git reset --hard origin/main` instead of `git pull`, which fetches only `main` and sidesteps the broken branch ref entirely.
