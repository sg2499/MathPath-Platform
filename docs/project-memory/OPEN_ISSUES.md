## Resolved Recently (2026-07-13, premature mock-exam auto-submit — investigated and fixed as PR #318, verified)

Students reported mock exams auto-submitting while the on-screen timer still showed minutes remaining. Investigated before any code changed. Backend deadline enforcement was confirmed correct (`expires_at = started_at + duration_seconds`, re-checked on every answer-save and page load). Root cause was the client-side countdown: `frontend/hooks/useAttemptTimer.ts` (shared by mock/assessment/DPS-practice attempt pages, unchanged since the initial commit) counted down via a plain `setTimeout` chain with no anchor to wall-clock time and no resync on tab-focus regain (React Query's `refetchOnWindowFocus` is globally off). A backgrounded tab (student switches apps, phone locks) freezes the display while the server's real deadline keeps advancing; the next answer-save correctly detects real expiry and submits, while the frozen display still showed time left — reproducible for any student who left the tab mid-attempt, on any timed activity type, which is why only some students hit it.

**Fix:** `useAttemptTimer.ts` now anchors to an absolute deadline and self-corrects every tick regardless of throttling, plus a `visibilitychange` listener that resyncs immediately and refetches the attempt from the server the moment the tab regains focus. Wired into all three attempt pages (mock, assessment, DPS/practice). Also fixed in the same pass: `.math-timer-critical`'s pulsing animation referenced the wrong keyframes name (`math-timer-critical` vs. `@keyframes mathTimerCritical`) so the sub-5-minute urgency pulse never actually ran — one-line name fix; and `TestTimer.tsx` had no dark-mode variants on its non-urgent state — added.

**Delivery:** PR #318, squash commit `fa246c0`, all 11 CI checks passed (`npm run typecheck && npm run build` run for real in the developer's terminal, clean, 41/41 routes). Frontend-only, no backend/schema touched. **Not yet independently browser-verified live** — see Active item above.

- **Round 9 — universal mock-completion gamification fix.** After a second live batch of students (Pragya Ghosh, Sampreeti Mohapatra, Tanay Gupta, Sayantan Biswas, +others) completed mock exams and got no XP/coins, no badges, no submission notifications (student, teacher, or admin), and a garbage leaderboard "time taken" value, root-caused and fixed all four, universally, for existing and future students:

  **Root cause (XP/coins/badges/notifications):** `competition_mock_attempt_service.py` had two code paths that could grade/complete a mock attempt without ever running its notification/economy/achievement side-effects, because those side-effects only lived in a separate wrapper function (`SubmitCompetitionMockAttemptForStudent`) that those paths bypassed entirely: (1) `EnsureCompetitionAttemptActiveOrSubmit()`, hit by any GET to load an attempt once the timer's already expired server-side (tab backgrounded, network dropped, or the student just reopens the page after time's up), and (2) the defensive result-summary repair in `_result_payload()`. Once an attempt's status flipped to completed via either path, the raw grading function's own "already completed" guard made it a permanent no-op for the real submit/auto-submit endpoints too — so the side-effects silently never ran, for however many students hit this (very plausible for most of a live class, since letting the timer run out or navigating away is common).

  **Fix:** relocated the notification/economy/achievement side-effects into the core grading function itself (`SubmitCompetitionMockAttempt`, now delegating to a new `_ProcessMockCompletionSideEffects`), guarded by a new `gamification_processed_at` timestamp column on `competition_mock_attempts` (Alembic migration `a1c9de3f7b21` + a matching `schema_migration.py` self-heal entry, per this repo's dual-safety convention) claimed atomically via a conditional `UPDATE ... WHERE gamification_processed_at IS NULL`. Every completion path — manual submit, explicit auto-submit, the lazy GET-triggered auto-submit, and the defensive result-page repair — now gets identical treatment, exactly once, with no dependency on which one happens to fire first. This is now live in production, so every future student and every future mock submission is covered automatically — nothing left to configure per-student.

The first created a *differently-typed* (`MOCK_EXAM_COMPLETED` vs the real `MOCK_SUBMITTED`) student-only notification with no teacher/admin coverage; the second did a full destructive wipe-and-rebuild of every student's badge/stat tables on every deploy, and never touched XP/coins at all. Both are now retired from startup (left in the codebase, just not called) in favor of the real fix plus a deliberate one-time backfill. Important downstream consequence discovered during backfill verification: because this old job ran on *every* backend restart for months, it had already organically sent most students their legitimate badge-unlock notifications over time — this is why the backfill's "badges newly unlocked" count (64) was much higher than its "notifications created" count (a handful) on the full recompute: most badges were being re-detected against a freshly-wiped table, not newly notified.
