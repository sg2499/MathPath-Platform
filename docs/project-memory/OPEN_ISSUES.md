- (2026-07-11, round 9 — applied, delivery deliberately withheld, same live-exam hold as rounds 6-8; universal fix, not scoped to specific students) After a second live batch of students (Pragya Ghosh, Sampreeti Mohapatra, Tanay Gupta, Sayantan Biswas, +others) completed mock exams and got no XP/coins, no badges, no submission notifications (student, teacher, or admin), and a garbage leaderboard "time taken" value, root-caused and fixed all four, universally, for existing and future students:

  **Root cause (XP/coins/badges/notifications):** `competition_mock_attempt_service.py` had two code paths that could grade/complete a mock attempt without ever running its notification/economy/achievement side-effects, because those side-effects only lived in a separate wrapper function (`SubmitCompetitionMockAttemptForStudent`) that those paths bypassed entirely: (1) `EnsureCompetitionAttemptActiveOrSubmit()`, hit by any GET to load an attempt once the timer's already expired server-side (tab backgrounded, network dropped, or the student just reopens the page after time's up), and (2) the defensive result-summary repair in `_result_payload()`. Once an attempt's status flipped to completed via either path, the raw grading function's own "already completed" guard made it a permanent no-op for the real submit/auto-submit endpoints too — so the side-effects silently never ran, for however many students hit this (very plausible for most of a live class, since letting the timer run out or navigating away is common).

  **Fix:** relocated the notification/economy/achievement side-effects into the core grading function itself (`SubmitCompetitionMockAttempt`, now delegating to a new `_ProcessMockCompletionSideEffects`), guarded by a new `gamification_processed_at` timestamp column on `competition_mock_attempts` (Alembic migration `a1c9de3f7b21` + a matching `schema_migration.py` self-heal entry, per this repo's dual-safety convention) claimed atomically via a conditional `UPDATE ... WHERE gamification_processed_at IS NULL`. Every completion path — manual submit, explicit auto-submit, the lazy GET-triggered auto-submit, and the defensive result-page repair — now gets identical treatment, exactly once, with no dependency on which one happens to fire first.

  **Also found and retired:** `main.py`'s `on_startup()` used to unconditionally run `ensure_mock_student_notifications_retroactive()` and `recalculate_all_gamification_stats()` on every single backend restart — a pre-existing, fragile attempt to paper over this same bug. The first created a *differently-typed* (`MOCK_EXAM_COMPLETED` vs the real `MOCK_SUBMITTED`) student-only notification with no teacher/admin coverage; the second did a full destructive wipe-and-rebuild of every student's badge/stat tables on every deploy (expensive, and itself a risk if the evaluation logic ever regresses), and never touched XP/coins at all. Both are now retired from startup (left in the codebase, just not called) in favor of the real fix plus a deliberate one-time backfill.

  **Also found and fixed (separate bug, same investigation):** the student dashboard's "Acquired XP / MathCoins" wallet widget never read the real economy ledger at all — it computed a locally-fabricated approximation (`completedAssignments*50 + completedAssessments*150 + badges*200`) that didn't even include mock exams in the formula, so it would never have moved regardless of the fix above. Added `GET /student/economy` (real `UserEconomy` read) and wired `frontend/app/student/dashboard/page.tsx`'s `StudentWallet` to it, removing the fake formula entirely (`currentLevel`/`xpIntoLevel` were unused dead code, also removed).

  **Also found and fixed (leaderboard, unrelated read-path bug):** both `get_mock_exam_leaderboard` and `get_cumulative_leaderboard` in `routes_student.py` computed the displayed "time taken" by rescaling `time_utilization_percentage` against a hardcoded 3600-second (60-minute) assumption instead of reading the real, already-correct `time_taken_seconds` column — producing fabricated values like "60m 0s" for a student who used 100% of a 30-minute exam. Fixed to read the real column directly; no backfill needed since the underlying data was always correct, only the leaderboard's display math was wrong.

  **Backfill:** `backend/scripts/diagnose_mock_gamification.py` (read-only, safe to run anytime, takes student names or `--all`) and `backend/scripts/backfill_mock_gamification.py` (`--dry-run` by default, `--apply` to write) — the latter gates economy awards and submission notifications per-attempt on whether they already exist (so already-correctly-processed attempts are never double-awarded), backdates backfilled notifications to the attempt's real `submitted_at`, and does a one-time deterministic full recompute of badges/stats (the same approach the now-retired startup job used to take automatically, just run once deliberately instead of on every restart). Neither script has been run yet — not yet delivered, and the backfill should only run once delivery happens and things are calm, not racing live traffic.

  Not yet verified against real production data (no DB access from this environment) — the diagnostic script is the deliberate first step for that once delivered.

- (2026-07-11, round 8 — applied, delivery deliberately withheld, same live-exam hold as round 7) Root-caused why the admin "Live Radar" widget (`/admin/dashboard`) always showed "No students active" even while students were actively taking mocks. Two frontend files — `frontend/components/admin/LiveRadarWidget.tsx` and `frontend/hooks/useHeartbeat.ts` (the latter mounted globally in `AppShell.tsx` for every logged-in page, specifically to keep `last_active_at` fresh during idle browsing) — both built their request URL from `process.env.NEXT_PUBLIC_API_URL`, a variable that is not defined anywhere in this project (`.env.example` and every other call site use `NEXT_PUBLIC_API_BASE_URL`). With the var undefined, both `fetch()` calls resolved to the literal relative path `"undefined/api/..."` against the frontend's own origin instead of the backend, 404'd, and were silently swallowed by their `catch` blocks — so the widget's `GET /admin/live-students` call never reached the backend (always empty list) and the heartbeat's `GET /auth/ping` never reached the backend either (only the debounced background task on other authenticated requests was keeping `last_active_at` fresh, which is fine while a student is actively clicking through a mock but not during idle reading/thinking time). Fixed both by switching to the shared `api` axios client (`frontend/lib/api.ts`), which already resolves the correct backend base URL from the correctly-named env var and attaches the auth token automatically — matching the pattern used everywhere else in the codebase. Backend side (`GET /api/admin/live-students` in `routes_admin.py`, `GET /api/auth/ping` in `routes_auth.py`, and the `last_active_at` update wiring in `dependencies.py`) was already correct and needed no changes. Not yet delivered/pushed — same hold as round 7, since Vercel/Render would redeploy mid-exam. Once delivered, this should immediately start reflecting real student counts.

## Resolved Recently

- 2026-07-11 (round 2, prepared not yet delivered): Two more real bugs found by looking at the live heatmap after round 1 shipped. (a) Bars weren't visually rendering at all — only tooltips — because the heatmap row used `items-end` on a flex container, leaving each day-column's own height as auto/content-based, so the bar's `height: X%` had no defined ancestor to resolve against and collapsed to zero. Fixed by giving each bar an explicit-height (`h-20`) track to size against. (b) The flowState formula rewarded raw speed (flat "5 questions/minute" threshold) over accuracy, so a 97%-accuracy/60-minute mock scored a lower tier than a 92%-accuracy/30-minute one — because the two mocks have very different intended durations and a flat threshold can't tell them apart. Fixed by scoring pace per attempt against that specific task's own admin-defined expected duration (new additive `expectedDurationSeconds` field on `/student/results` rows and both `attemptHistory` payloads) instead of one global number; overall accuracy/speed weighting balance kept the same, since speed is intentionally a major factor in this abacus program. Also added an info-icon popover on the heatmap heading explaining what it tracks and what drives the tier, since students previously had no way to know.
- 2026-07-11 (round 1): Grind heatmap was silently undercounting activity — it only read practice-sheet `Results`, never mock exam or assessment-engine completions, even though both were already fetched by the dashboard page. Fixed by adding an additive `attemptHistory` array (full completed-attempt history, not just latest) to the mock-assignment and assessment-assignment backend payloads, and rewriting the dashboard's `grindData` calculation to pool practice + mock + assessment events per calendar day before computing one flowState/tier per day.
