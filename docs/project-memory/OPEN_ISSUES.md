- **Round 9 — universal mock-completion gamification fix.** After a second live batch of students (Pragya Ghosh, Sampreeti Mohapatra, Tanay Gupta, Sayantan Biswas, +others) completed mock exams and got no XP/coins, no badges, no submission notifications (student, teacher, or admin), and a garbage leaderboard "time taken" value, root-caused and fixed all four, universally, for existing and future students:

  **Root cause (XP/coins/badges/notifications):** `competition_mock_attempt_service.py` had two code paths that could grade/complete a mock attempt without ever running its notification/economy/achievement side-effects, because those side-effects only lived in a separate wrapper function (`SubmitCompetitionMockAttemptForStudent`) that those paths bypassed entirely: (1) `EnsureCompetitionAttemptActiveOrSubmit()`, hit by any GET to load an attempt once the timer's already expired server-side (tab backgrounded, network dropped, or the student just reopens the page after time's up), and (2) the defensive result-summary repair in `_result_payload()`. Once an attempt's status flipped to completed via either path, the raw grading function's own "already completed" guard made it a permanent no-op for the real submit/auto-submit endpoints too — so the side-effects silently never ran, for however many students hit this (very plausible for most of a live class, since letting the timer run out or navigating away is common).

  **Fix:** relocated the notification/economy/achievement side-effects into the core grading function itself (`SubmitCompetitionMockAttempt`, now delegating to a new `_ProcessMockCompletionSideEffects`), guarded by a new `gamification_processed_at` timestamp column on `competition_mock_attempts` (Alembic migration `a1c9de3f7b21` + a matching `schema_migration.py` self-heal entry, per this repo's dual-safety convention) claimed atomically via a conditional `UPDATE ... WHERE gamification_processed_at IS NULL`. Every completion path — manual submit, explicit auto-submit, the lazy GET-triggered auto-submit, and the defensive result-page repair — now gets identical treatment, exactly once, with no dependency on which one happens to fire first. This is now live in production, so every future student and every future mock submission is covered automatically — nothing left to configure per-student.

  **Also found and retired:** `main.py`'s `on_startup()` used to unconditionally run `ensure_mock_student_notifications_retroactive()` and `recalculate_all_gamification_stats()` on every single backend restart — a pre-existing, fragile attempt to paper over this same bug. The first created a *differently-typed* (`MOCK_EXAM_COMPLETED` vs the real `MOCK_SUBMITTED`) student-only notification with no teacher/admin coverage; the second did a full destructive wipe-and-rebuild of every student's badge/stat tables on every deploy, and never touched XP/coins at all. Both are now retired from startup (left in the codebase, just not called) in favor of the real fix plus a deliberate one-time backfill. Important downstream consequence discovered during backfill verification: because this old job ran on *every* backend restart for months, it had already organically sent most students their legitimate badge-unlock notifications over time — this is why the backfill's "badges newly unlocked" count (64) was much higher than its "notifications created" count (a handful) on the full recompute: most badges were being re-detected against a freshly-wiped table, not newly notified.

  **Also found and fixed (separate bug, same investigation):** the student dashboard's "Acquired XP / MathCoins" wallet widget never read the real economy ledger at all — it computed a locally-fabricated approximation that didn't even include mock exams. Added `GET /student/economy` (real `UserEconomy` read) and wired `frontend/app/student/dashboard/page.tsx`'s `StudentWallet` to it.

  **Also found and fixed (leaderboard, unrelated read-path bug):** both `get_mock_exam_leaderboard` and `get_cumulative_leaderboard` in `routes_student.py` computed the displayed "time taken" by rescaling `time_utilization_percentage` against a hardcoded 3600-second (60-minute) assumption instead of reading the real, already-correct `time_taken_seconds` column. Fixed to read the real column directly; no backfill needed since the underlying data was always correct.

  **Backfill — run to completion and verified clean.** `backend/scripts/diagnose_mock_gamification.py` (read-only) and `backend/scripts/backfill_mock_gamification.py` (`--dry-run`/`--apply`) were run from the developer's local terminal against production. `--apply` ran twice: 54 historical attempts scanned across 9 students; first pass awarded missing XP/coins (6000 XP / 285 coins total) and created missing notifications; badge/stat recompute unlocked 64 badges (of which only 3 needed a fresh notification — see above for why). A real gap was found afterward: the script's original notification-dedup logic gated ALL notification creation (student+teacher+admin) on one boolean check ("does the student's MOCK_SUBMITTED notif exist"), which missed a genuine historical partial state — Meera Chatterjee's `MM_17` attempt had student+teacher notified but its admin notification was silently missing from whatever code ran before this was ever fixed. Fixed the script to check student, teacher, and each individual active admin independently; re-ran `--apply`, closed the gap. Final full-platform sweep via `diagnose_mock_gamification.py --all` (not scoped to the originally-reported students) confirms **zero remaining gaps across all 9 students and all 54 completed mock attempts** — every attempt shows OK for XP/coins, student notif, teacher notif, and admin notifs. This partial-notification failure mode cannot recur going forward: the live production hook is atomic per-attempt (one try block, one commit for all three recipient types together), so it was only ever possible in pre-fix historical data.

  Also fixed two script-quality issues found while running the backfill (not product bugs): the dry-run summary undercounted notifications created (fixed a counting bug), and the script went silent for minutes during the badge recompute with no way to distinguish "working" from "hung" (added per-student progress printing + forced `print(..., flush=True)` since Windows terminals were buffering output during the long loop).
