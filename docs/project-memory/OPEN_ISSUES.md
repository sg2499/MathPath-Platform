## Resolved Recently (2026-07-18 round 2, trivial-operand bug in MM + IM multiplication/division/BODMAS generators — CONFIRMED DELIVERED AND LIVE as PR #345, `2d414db`)

**Delivery + verification, added round 4 (same day):** Shailesh committed/pushed as PR #345 (`2d414db`), confirmed via `git log`/`git fetch origin main` matching `origin/main`, `git diff --stat 2c6e0dc 2d414db` confirming the diff is scoped to exactly `im/operands.py` and `mm/operands.py`, and Render/Vercel deploy-dashboard screenshots showing `2d414db` Live/Ready in Production. Independently re-verified live via Claude-in-Chrome against the real production admin panel (not just sandbox tests): IM-L2 multiplication regenerated twice, 0 trivial operands; IM-L2 BODMAS regenerated, 0 trivial embedded divisors; MM's BODMAS (MM-L1 Lesson 19 DPS4) regenerated live, 0 real trivial hits (one scan false-positive on a `×10%` percentage term — correctly excluded, since percentage terms are a different concept family this fix deliberately left untouched — corrected and re-run clean).

Shailesh flagged a live admin-preview screenshot: IM-L2 multiplication questions like "9663 x 1 = ?" — a student can answer without computing, since anything x 1 is itself. Asked that this be checked for both multiplication AND division, across every level in both MM and IM, plus that MCQ distractors not be guessable by pattern alone.

**Root cause (IM):** two independent bugs in `im/operands.py` — `_IsTrivial()` only flagged exact multiples of 100 >= 100 (never caught a literal 1), and `GenerateWholeNumberMultiplication()`'s retry loop only re-rolled when BOTH operands were trivial (AND, not OR), so a single trivial 1-digit side was accepted on the first attempt every time. Fixed by porting MM's existing, correct `_IsTrivialScaleOperand` definition (0, 1, or any value whose only nonzero digit is the leading one: 10, 20, 50, 100, 300, 1000, ...) into IM, fixing the AND->OR logic, and extending the guard to `GenerateWholeNumberDivision`'s divisor (previously only excluded literal 1) and to embedded division/multiplication terms inside both IM BODMAS templates (L2/L3's exact-division shape and L4's own basic/squared-term shape, the latter having zero guard at all on its embedded terms).

**Root cause (MM):** MM's *standalone* multiplication/division generators were already fully protected via the same `_IsTrivialScaleOperand` guard (confirmed via audit, no bug there). But MM's BODMAS composite-expression family (6 stage-specific builders, reachable across lessons 1, 2, 4, 6, 10, 11, 16, 19, 21, 22, 23, 27 — essentially every lesson band with a BODMAS section) had the identical unguarded-embedded-term gap in 5 places, plus a hand-picked `BracketDivisor` choice list that was majority-trivial by design (10, 20, 40, 50, 100 of 7 choices). All fixed with the same guard already used elsewhere in that file.

**Deliberately not touched:** percentage-specific round numbers (10%, 20%, 25%, 50%, etc., used in several BODMAS percent terms) — a different concept family from the "multiplication/division sum" complaint, and workbook-native convention that shouldn't change without being asked.

**MCQ distractor quality:** audited separately, confirmed already solid — the shared `smart_distractors.py` engine (used by both MM and IM) already enforces last-digit matching, a plausibility magnitude band, and operation-specific "genuine mistake" candidate pools (digit-transpose, middle-digit-shift, missed-row, sign-flip). Not the cause of the reported bug and needed no changes.

**Verified:** 24/24 existing pytest pass. IM: 40,100 standalone mult/div questions across all 3 levels, 0 trivial operands; 2,700 BODMAS questions across all 3 levels, 0 trivial embedded divisors; full 60-DPS stress test, 16,275 questions, 0 errors (no regression). MM: 2,400 BODMAS questions across all 12 affected lessons, 0 trivial embedded operands; full 30-lesson stress test, 7,830 questions, 0 errors (no regression). **Delivered and live — see round 4 verification note above.**
