## Resolved Recently (2026-07-11 round 3, prepared not yet delivered)

- After round 2 shipped, the developer's own screenshots caught three more real issues: (a) the info popover rendered as a small anchored dropdown that visually overlapped the bars underneath it — the card uses 3D transforms for its tilt-hover effect, which changes how nested absolute/fixed positioning stacks, so it was tangling with the bars instead of sitting cleanly on top; fixed by making it a dedicated overlay scoped to just the card's own boundary (centered within the card, not the whole screen — confirmed with the developer that page-wide centering would feel disconnected). (b) All bars were the same color regardless of tier — fixed by giving each tier (S/A/B/C/D) its own distinct bar color, gray for rest days. (c) Clicking the info icon a second time to close it was also flipping the whole card to the quote-display side — root cause was the close-on-click-outside handler not calling stopPropagation, so the click kept bubbling to the card's flip handler after closing the popover; fixed by stopping propagation at every layer (trigger, backdrop, and popover content).

## Resolved Recently

- 2026-07-11 (round 2, prepared not yet delivered): Two more real bugs found by looking at the live heatmap after round 1 shipped. (a) Bars weren't visually rendering at all — only tooltips — because the heatmap row used `items-end` on a flex container, leaving each day-column's own height as auto/content-based, so the bar's `height: X%` had no defined ancestor to resolve against and collapsed to zero. Fixed by giving each bar an explicit-height (`h-20`) track to size against. (b) The flowState formula rewarded raw speed (flat "5 questions/minute" threshold) over accuracy, so a 97%-accuracy/60-minute mock scored a lower tier than a 92%-accuracy/30-minute one — because the two mocks have very different intended durations and a flat threshold can't tell them apart. Fixed by scoring pace per attempt against that specific task's own admin-defined expected duration (new additive `expectedDurationSeconds` field on `/student/results` rows and both `attemptHistory` payloads) instead of one global number; overall accuracy/speed weighting balance kept the same, since speed is intentionally a major factor in this abacus program. Also added an info-icon popover on the heatmap heading explaining what it tracks and what drives the tier, since students previously had no way to know.
- 2026-07-11 (round 1): Grind heatmap was silently undercounting activity — it only read practice-sheet `Results`, never mock exam or assessment-engine completions, even though both were already fetched by the dashboard page. Fixed by adding an additive `attemptHistory` array (full completed-attempt history, not just latest) to the mock-assignment and assessment-assignment backend payloads, and rewriting the dashboard's `grindData` calculation to pool practice + mock + assessment events per calendar day before computing one flowState/tier per day. Delivered to main as PR #303.
