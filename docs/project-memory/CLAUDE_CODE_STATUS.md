- **Root cause found (via live Render log capture, timed with a fresh request so it appeared in Live Tail before free-tier retention could drop it):** `sqlalchemy.exc.IntegrityError: null value in column "lesson_id" of relation "assessment_questions" violates not-null constraint`. The migration that makes `assessment_questions.lesson_id` nullable (needed because section-wise questions span multiple lessons, so they have no single lesson to point at) never actually applied to production. Digging further (Shailesh ran `alembic upgrade head` from his local machine against production's External DB URL) revealed *why*: production's database has **never been tracked by Alembic at all** — `alembic upgrade head` tried to replay the entire migration history from `18fa414af0ad` (the very first migration) and immediately hit `DuplicateTable` errors, because every table in production was actually created by `Base.metadata.create_all()` plus the custom `ensure_*_column()` startup functions in `schema_migration.py`, not by Alembic. This is a real, structural gap worth its own follow-up (see below) — Alembic migrations in `alembic/versions/` are effectively decorative for this production database right now, `render.yaml`'s `alembic upgrade head` build step has likely never done anything real.
- **Immediate fix applied and verified:** Shailesh ran a one-off direct `ALTER TABLE assessment_questions ALTER COLUMN lesson_id DROP NOT NULL` against production via a local Python script (`engine.begin()` + raw SQL), using the External Database URL. Confirmed safe (loosening NOT NULL never touches existing data) and confirmed working: immediately after, both MM-L1 and IM-L4 section-wise assessment creation succeeded live in production — `POST /api/admin/assessment-blueprints` returned 200, `GET .../generated-assessment` confirmed MM-L1 (40 questions, 40 marks flat, 10 real sections) and IM-L4 (40 questions, 60 marks = 35×1 + 5×5 for Skill Stacker/Concept Drill, 8 real sections) both exactly correct.
- **Follow-up worth doing separately (not blocking, not started):** decide whether to (a) properly initialize Alembic's `alembic_version` tracking on production by stamping it to the current actual schema state (`alembic stamp head` after confirming the stamped revision truly matches live schema), so future migrations can rely on `alembic upgrade head` actually working, or (b) keep using the `ensure_*_column()` pattern in `schema_migration.py` for schema changes (matching how this codebase has evidently always actually evolved production) and stop relying on `render.yaml`'s alembic build step / stop writing new Alembic migration files that will never apply. Whichever direction, the current state (an alembic migration directory that looks authoritative but isn't) is a trap for the next schema change — flag this to Shailesh as its own conversation.
