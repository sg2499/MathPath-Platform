## 2026-07-15 update (round 2): IM "Long Division & Estimation" concept fix ŌĆö prepared, not yet delivered

**The ask:** Shailesh flagged that IM Level 4's "Long Division & Estimation" division sections (3D├Ę1D and 4D├Ę1D, both Abacus and Visual) were generating plain exact division (e.g. "9115 ├Ę 5 = 1823") ŌĆö no different from a regular division concept. His spec: these questions should have a genuine remainder, and the answer should be the quotient computed to 3 decimal places (truncated, not rounded ŌĆö the literal next long-division digit), then rounded to 2 decimal places on that 3rd decimal digit (< 5 rounds down, >= 5 rounds up). Worked example given: 415 ├Ę 7 = 59.285714... ŌåÆ truncate to 59.285 ŌåÆ 3rd decimal is 5 ŌåÆ rounds up to 59.29. Must apply across DPS, assessments, and competition mocks, and be built generically so any future level with the same concept picks it up automatically.

**Investigation first, confirmed the scope exactly matches Shailesh's description before touching anything:** IM's curriculum map (`curriculum_map.py`'s `_DIV()` helper) already has a distinct `isLongDivisionEstimation` flag, set on exactly the 3D/1D and 4D/1D division sections (both display variants) ŌĆö matching Shailesh's examples precisely; other division patterns (3D/2D, 4D/2D, 4D/3D) are plain "Division" and correctly untouched. The bug: `GenerateWholeNumberDivision()` (`backend/app/question_engine/im/operands.py`) read this flag into output metadata but never used it to change the actual number generation ŌĆö it always picked a divisor then multiplied by a quotient to construct a guaranteed-exact dividend, regardless of the flag. One single shared function, already reused identically by DPS practice, assessments, and competition mocks (`GenerateImQuestionSet()` is the one call site for all three) ŌĆö so fixing it here fixes all three delivery contexts at once, and it's already keyed off a config flag rather than hardcoded to IM-L4, so a future level just needs to set the same flag.

**Fix:**
- `TruncateThenRoundQuotient()` (new, exported) ŌĆö implements the exact rounding rule from the worked example: quantize to 3 decimals with `ROUND_DOWN` (truncate), then quantize that to 2 decimals with `ROUND_HALF_UP`.
- `_GenerateLongDivisionEstimationPair()` (new) ŌĆö digit-pattern-agnostic dividend/divisor generator that rejects any pair dividing evenly (per Shailesh's explicit call: always force a genuine remainder, never allow an accidental exact result).
- `GenerateWholeNumberDivision()` ŌĆö now branches on `isLongDivisionEstimation`: the long-division branch uses the two functions above; the plain-division branch is untouched (byte-identical behavior, verified by re-reading the diff).
- `_ValidateDivision()` (`validators.py`) ŌĆö now takes `Config` (matching the existing pattern `_ValidateAnswerPosition()` already used) so it can check the same two rules: reject an exact-division pair, and require the answer to match `TruncateThenRoundQuotient()` exactly, when the flag is set.
- Distractor generation needed **no changes** ŌĆö confirmed by reading `smart_distractors.py`: every distractor helper derives its decimal precision from `correct_answer.as_tuple().exponent`, the same generic mechanism already proven for decimal Add/Less and BODMAS answers, so it picks up the new 2-decimal-place answers automatically.
